# JWT 設計壁打ちスキル

ユーザーが新しい JWT を設計・定義する際に、対話的にガイドする。

以下の知識に基づいてユーザーの壁打ち相手になること。

## 共通鍵方式（HS256等） vs 公開鍵方式（RS256等）

共有秘密鍵方式では署名側と検証側が同じ鍵を保持するため、検証側が署名を偽造できるリスクがある。ただし、署名側・検証側がともに同一組織の管理下にあるサービスである場合、このリスクは実質的に問題とならない。

将来的に外部サービスへ検証を委ねるユースケースが想定されない限り、公開鍵方式（RS256等）を採用する必要はなく、共有秘密鍵方式（HS256等）で十分である。

## HS256 vs HS512

基本的に HS256 で十分。

- HS256 は広く利用されているアルゴリズムであり、署名強度として十分である
- RFC 7518 Section 3.1 において HS256 は **"Required"（実装必須）** と定められており、どの JWT ライブラリでも実装されている
  - https://datatracker.ietf.org/doc/html/rfc7518#section-3.1

## HS256 の鍵長要件

- RFC 7518 において、HMAC アルゴリズムではハッシュ出力と同じサイズ以上の鍵を使用しなければならない（**MUST**）と定められている
  - HS256 の場合: **最低 256 ビット（32 バイト）**
- Auth0 は 512 ビットの鍵を採用しており、これに合わせると安心
  - https://auth0.com/blog/brute-forcing-hs256-is-possible-the-importance-of-using-strong-keys-to-sign-jwts/
